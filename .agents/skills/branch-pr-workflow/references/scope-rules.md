# Commit Scope Rules

## 포함 기준
- 현재 사용자 요청을 직접 해결하는 변경만 포함한다.
- 테스트 변경은 해당 기능 변경과 직접 연결될 때 포함한다.
- 문서 변경은 동작/아키텍처 계약이 바뀌는 경우 포함한다.

## 제외 기준
- 현재 PR 목적과 무관한 리팩터링/포맷팅은 제외한다.
- 다른 기능 작업 중 우연히 수정된 파일은 제외한다.
- 대용량 lockfile 변경은 실제 의존성 변경이 있을 때만 포함한다.

## 커밋 분리 기준
- 기능 구현, 하네스/테스트 보강, 문서 업데이트는 필요 시 분리한다.
- 롤백이 독립적으로 가능하도록 논리 단위를 유지한다.
- 커밋 메시지는 `무엇 + 왜`를 한 줄로 설명 가능해야 한다.

## 브랜치 분리 기준
- 기능 목적이 다르면 브랜치를 분리한다.
- 리뷰어가 한 번에 이해하기 어려운 범위(파일/영역 과다)면 새 브랜치로 분리한다.
